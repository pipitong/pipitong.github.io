从零搭建线上数据管道：需求、架构与表设计

2025-09 · 约 5 分钟 · 架构 · Prefect · ClickHouse · 从零搭建

记录在一家量化公司从零参与搭建线上数据管道的思路：从需求与目标到数据源、架构选型、表设计、可靠性设计，再到上线与迭代。

1. 需求与目标

先明确业务要什么：量化研究需要哪些数据、更新频率、历史深度、一致性要求。例如：多交易所 K 线/行情/资金费率/持仓量，部分链上或宏观数据；实时或准实时（分钟级/小时级）；需要能回溯历史、支持离线补数。目标可以概括为：多源、及时、完整、可追溯。

2. 数据源与更新频率

列出所有数据源（如 Binance、Bybit、OKX、Glassnode 等），每个接口的 rate limit、推荐拉取间隔、数据量级。这一步直接决定后面调度策略和存储规划。不同交易所 API 差异大，需要统一抽象成「数据源 + endpoint + 更新频率」便于编排。

3. 架构选型：调度与存储

调度与编排：我们选用 Prefect，每个 endpoint 对应一个 flow，按 cron/interval 调度，便于限流和独立重跑。存储：ClickHouse 作为统一分析型存储，按库/表做分区；原始数据可先落 S3 再入 ClickHouse，或直接写库，视延迟与成本权衡。

4. 表设计与分区

按业务域建表（如 kline、funding_rate、open_interest），主键与分区键设计要兼顾写入与查询（常见按时间 + 交易所/交易对）。注意 TTL 与保留策略，避免存储无限膨胀。

5. 限流、重试与监控

每个 flow 内对 API 调用做 rate limit，失败时重试 + backoff；保证「上游拉取成功后，下游写入通过重试完成」，避免半截数据。用 Prefect 的 run 状态和日志做监控与告警，能快速定位是哪个 endpoint/flow 出问题。

6. 上线与迭代

先上核心数据源与核心表，再逐步扩展；历史数据用离线任务补全，与在线流在 schema 和幂等写入上保持一致。后续可迭代：更多数据源、更细粒度监控、数据质量校验与告警等。

这套流程的核心原则：数据源→调度→存储每一层都要可独立重跑，历史补数和线上采集共用同一套 schema 与幂等写入逻辑。具体细节（表结构、限流策略）在系列后续文章里展开。
