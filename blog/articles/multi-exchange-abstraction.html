统一接多个交易所：Endpoint/Reader 抽象层的设计

2025-10 · 约 11 分钟

接第二个交易所时，我们复制了第一个的大部分代码；接第三个时，发现修一个 bug 要改三个地方。第四个之前，决定重新设计接入层。

这篇记录最终的抽象方式——Endpoint / Reader 两层设计，以及这个设计在接入 Hyperliquid（API 风格和 Binance 完全不同）时是否经得住考验。

问题：每个交易所「差一点但又差很多」

表面上看，各交易所的 API 很像：都是 HTTP 请求，都返回 JSON，都有分页。实际工程里差异很多：

维度 Binance OKX KuCoin Hyperliquid
HTTP 方法 GET GET GET POST (全部)
分页方式 startTime 升序 after/before 降序 startAt 升序（秒） startTime 升序（但参数在 body）
时间戳单位 ms ms s ms
限流单位 weight / min req/s per endpoint weight / 30s req/s 全局
K 线字段格式 数组 [ts, o, h, l, c, v, ...] 数组 [ts, o, h, l, c, vol, ...] 数组但字段顺序不同 对象 {"t": ..., "o": ...}

这些差异分散在代码里，就是每个函数里一堆 if exchange == "okx":。接到第三个交易所时，这种方式的维护成本已经超过了当初省下的设计时间。

两层抽象：Endpoint 和 Reader

拆分职责：

  • Endpoint：「这个数据在这个交易所怎么取」—— API 路径、请求参数构造、响应解析、分页逻辑、这次请求消耗多少 rate limit weight
  • Reader：「取数据这件事本身的通用逻辑」—— HTTP 客户端、限流控制、重试、错误处理、并发协调

Reader 不关心取的是哪个交易所、哪种数据类型——它只知道「给我一个 Endpoint，我来负责发请求、限流、重试、返回结果」。

Endpoint 基类

Endpoint 基类 
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any, AsyncIterator

@dataclass
class RawPage:
    """一次 API 请求的原始返回结果。"""
    data: Any
    has_more: bool
    next_cursor: Any = None   # 用于下一次请求的 cursor（时间戳或 ID）

class Endpoint(ABC):
    """
    描述「如何从某个交易所取某种数据的一页」。
    子类只需实现三个方法。
    """

    @property
    @abstractmethod
    def rate_limit_weight(self) -> int:
        """这次请求消耗多少 weight（用于限流控制）。"""

    @abstractmethod
    def build_request(self, cursor: Any = None) -> dict:
        """
        构造 HTTP 请求参数。
        返回: {"method": "GET", "url": "...", "params": {...}}
        或:   {"method": "POST", "url": "...", "json": {...}}
        """

    @abstractmethod
    def parse_response(self, raw: Any) -> RawPage:
        """
        解析响应，提取数据行和分页信息。
        raw: 已 JSON 解析的响应体
        """

    def normalize_row(self, row: Any) -> dict:
        """
        可选：把 exchange-specific 格式转成统一 schema。
        默认直接返回（子类可 override）。
        """
        return row

Reader：通用执行层

Reader 基类（处理限流、重试、分页） 
import asyncio
import aiohttp
from typing import AsyncIterator

class Reader:
    def __init__(self, rate_limiter, session: aiohttp.ClientSession):
        self._limiter = rate_limiter
        self._session = session

    async def fetch_page(self, endpoint: Endpoint, cursor=None) -> RawPage:
        """发一次请求，处理限流和重试。"""
        request = endpoint.build_request(cursor)
        weight = endpoint.rate_limit_weight

        for attempt in range(3):
            await self._limiter.acquire(weight)
            try:
                async with self._session.request(**request) as resp:
                    if resp.status == 429:
                        retry_after = int(resp.headers.get("Retry-After", 30))
                        await asyncio.sleep(retry_after)
                        continue
                    resp.raise_for_status()
                    raw = await resp.json()
                    return endpoint.parse_response(raw)
            except aiohttp.ClientError as e:
                if attempt == 2:
                    raise
                await asyncio.sleep(2 ** attempt)

    async def paginate(self, endpoint: Endpoint, start_cursor=None) -> AsyncIterator[list[dict]]:
        """翻页直到没有更多数据，每页 yield 一批标准化的行。"""
        cursor = start_cursor
        while True:
            page = await self.fetch_page(endpoint, cursor)
            rows = [endpoint.normalize_row(r) for r in page.data]
            if rows:
                yield rows
            if not page.has_more:
                break
            cursor = page.next_cursor
            await asyncio.sleep(0.05)   # 对交易所友好

实现示例：Binance K 线

BinanceFuturesKlineEndpoint 
class BinanceFuturesKlineEndpoint(Endpoint):
    def __init__(self, symbol: str, bar: str, limit: int = 1000):
        self.symbol = symbol
        self.bar = bar
        self.limit = limit

    @property
    def rate_limit_weight(self) -> int:
        # Binance: limit <= 100 → weight 1, <= 500 → weight 2, <= 1000 → weight 5
        if self.limit <= 100:
            return 1
        elif self.limit <= 500:
            return 2
        return 5

    def build_request(self, cursor=None) -> dict:
        params = {
            "symbol": self.symbol,
            "interval": self.bar,
            "limit": self.limit,
        }
        if cursor is not None:
            params["startTime"] = cursor   # Binance: 升序分页用 startTime
        return {
            "method": "GET",
            "url": "https://fapi.binance.com/fapi/v1/klines",
            "params": params,
        }

    def parse_response(self, raw: list) -> RawPage:
        if not raw:
            return RawPage(data=[], has_more=False)
        return RawPage(
            data=raw,
            has_more=len(raw) == self.limit,
            next_cursor=raw[-1][0] + 1,   # 下一页从最后一根 K 线的时间戳 + 1ms 开始
        )

    def normalize_row(self, row: list) -> dict:
        return {
            "open_time": row[0],
            "open":      float(row[1]),
            "high":      float(row[2]),
            "low":       float(row[3]),
            "close":     float(row[4]),
            "volume":    float(row[5]),
            "quote_volume": float(row[7]),
            "trades":    int(row[8]),
        }

OKX 的差异在哪里

OKX K 线接口默认降序返回（最新的在前），分页靠 after（比这个时间戳旧）。这个逻辑完全封装在 OKX 的 Endpoint 子类里，Reader 无感知：

OKXKlineEndpoint（关键差异部分） 
class OKXKlineEndpoint(Endpoint):
    def build_request(self, cursor=None) -> dict:
        params = {
            "instId": self.symbol,
            "bar": self.bar,
            "limit": str(self.limit),
        }
        if cursor is not None:
            params["after"] = str(cursor)   # OKX: 降序分页，取比 cursor 更旧的数据
        return {
            "method": "GET",
            "url": "https://www.okx.com/api/v5/market/candles",
            "params": params,
        }

    def parse_response(self, raw: dict) -> RawPage:
        rows = raw.get("data", [])
        if not rows:
            return RawPage(data=[], has_more=False)
        return RawPage(
            data=rows,
            has_more=len(rows) == self.limit,
            next_cursor=rows[-1][0],   # 最后一条（最旧的）时间戳作为下一页的 after
        )

    def normalize_row(self, row: list) -> dict:
        return {
            "open_time":    int(row[0]),
            "open":         float(row[1]),
            "high":         float(row[2]),
            "low":          float(row[3]),
            "close":        float(row[4]),
            "volume":       float(row[5]),
            "quote_volume": float(row[7]),
        }

Hyperliquid：POST-only API 的处理

Hyperliquid 是全 POST 的 API，请求体是 JSON，这在 Endpoint 里是完全合法的——只需要把 build_request 返回 POST 格式即可，Reader 层不需要任何修改：

HyperliquidKlineEndpoint（POST 风格） 
class HyperliquidKlineEndpoint(Endpoint):
    BASE_URL = "https://api.hyperliquid.xyz/info"

    def __init__(self, symbol: str, bar: str, limit: int = 500):
        self.symbol = symbol
        self.bar = bar
        self.limit = limit

    @property
    def rate_limit_weight(self) -> int:
        return 1   # Hyperliquid: 固定 1 req 计一次

    def build_request(self, cursor=None) -> dict:
        body = {
            "type": "candleSnapshot",
            "req": {
                "coin": self.symbol,
                "interval": self.bar,
                "startTime": cursor or 0,
                "endTime": int(time.time() * 1000),
            }
        }
        return {
            "method": "POST",
            "url": self.BASE_URL,
            "json": body,    # aiohttp 的 json= 参数自动 Content-Type: application/json
        }

    def parse_response(self, raw: list) -> RawPage:
        if not raw:
            return RawPage(data=[], has_more=False)
        return RawPage(
            data=raw,
            has_more=len(raw) == self.limit,
            next_cursor=raw[-1]["t"] + 1,
        )

    def normalize_row(self, row: dict) -> dict:
        return {
            "open_time": row["t"],
            "open":      float(row["o"]),
            "high":      float(row["h"]),
            "low":       float(row["l"]),
            "close":     float(row["c"]),
            "volume":    float(row["v"]),
        }

Exchange Registry：统一注册与查找

所有 Endpoint 子类注册到一个 registry，调用方只需要传 exchange + data_type + symbol，不需要知道用哪个子类：

Endpoint registry 
from typing import Type

_REGISTRY: dict[tuple[str, str], Type[Endpoint]] = {}

def register(exchange: str, data_type: str):
    """装饰器：把 Endpoint 子类注册到全局 registry。"""
    def decorator(cls: Type[Endpoint]) -> Type[Endpoint]:
        _REGISTRY[(exchange, data_type)] = cls
        return cls
    return decorator

def get_endpoint_class(exchange: str, data_type: str) -> Type[Endpoint]:
    key = (exchange, data_type)
    if key not in _REGISTRY:
        raise ValueError(f"No endpoint registered for {exchange}/{data_type}")
    return _REGISTRY[key]

# 使用方式：
@register("binance_futures", "kline")
class BinanceFuturesKlineEndpoint(Endpoint):
    ...

@register("okx", "kline")
class OKXKlineEndpoint(Endpoint):
    ...

# 调用方：
EndpointCls = get_endpoint_class("hyperliquid", "kline")
endpoint = EndpointCls(symbol="BTC", bar="1m")
async for batch in reader.paginate(endpoint, start_cursor=start_ts):
    await write_to_clickhouse(batch)

效果：新增交易所的工作量变化

重构前后对比：

重构前 重构后
新增一个交易所的 K 线采集 复制现有代码 → 改 API URL、参数、分页逻辑、限流逻辑：约 1–2 天 实现 Endpoint 子类 3 个方法：约 2–4 小时
修复限流 bug（如超配额） 每个交易所各改一遍 只改 Reader 层，一次修全部交易所
新增一种数据类型（如 OI） 每个交易所各写一套 每个交易所实现一个 OI Endpoint，Reader 复用
单元测试覆盖 难以 mock，因为 API 调用分散在各处 只需 mock build_request 和 parse_response，Reader 层可独立测试

什么时候这个抽象会「漏」

Endpoint/Reader 抽象不是万能的，有几个地方需要打补丁：

  • 认证方式差异：有些交易所需要在 header 里签名（Binance、OKX），有些需要 API key 作为 query param（KuCoin），有些公开接口不需要任何认证（Hyperliquid 大部分数据）。认证逻辑放在 Reader 的子类里（如 BinanceReader、OKXReader），不进 Endpoint，因为认证是「和交易所建立连接的方式」，而不是「取某种数据的逻辑」。
  • OKX 的历史 K 线需要切换接口路径：数据全景那篇提到了这个问题——超过 1440 根要换 endpoint。这个逻辑放在 OKXKlineEndpoint 里，通过 build_request 动态决定用哪个 URL。
  • WebSocket 数据：上面的抽象是 REST 的。WebSocket 实时数据有不同的连接维护、消息解析、重连逻辑，需要另一套 StreamEndpoint 基类，不强行套进同一个 Endpoint 接口。
