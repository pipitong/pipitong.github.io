ClickHouse 在量化数据场景的选型与实践

2025-12 · 约 10 分钟

量化数据的读写模式很有特点：写入是持续批量追加（每分钟写一批 K 线），读取是大范围时间扫描（研究员查半年的资金费率或跑回测）。这两种模式加在一起，行式数据库的表现很差。这篇文章记录选 ClickHouse 的原因、核心表设计，以及实际使用中遇到的细节。

为什么选 ClickHouse

最初用 PostgreSQL。500M 行 K 线数据，一个简单的时间范围聚合查询（SELECT avg(close) FROM kline WHERE exchange='binance' AND symbol='BTCUSDT' AND open_time > '2024-01-01'）要跑 30 秒以上，加了索引和分区之后降到 8 秒，但仍然不够快。

评估了几个替代方案：

方案 优点 缺点
TimescaleDB 兼容 Postgres，迁移成本低 本质还是行存，压缩有改善但扫描大数据量仍慢
InfluxDB 为时序数据优化，写入快 不支持 JOIN；查询语言（Flux）学习成本高；多维标签场景下性能不稳定
ClickHouse 列存 + 向量化执行，大范围扫描快几个数量级；支持标准 SQL；压缩比高 不支持真正的 UPDATE/DELETE（需要用 ReplacingMergeTree）；点查（单行查询）不擅长
DuckDB 本地嵌入式，零运维；可直接查 Parquet 无服务端，不适合多用户并发查询；无持久化服务层

量化场景的查询几乎全是大范围扫描（「查 BTC 过去 6 个月所有 1m K 线」），极少单行点查，ClickHouse 是最合适的选择。同样的聚合查询，迁移到 ClickHouse 后从 8 秒降到 0.3 秒。

ReplacingMergeTree：解决幂等写入问题

量化数据需要频繁「补写」——历史补数、接口重试、数据修正都会把同一条数据写多次。ClickHouse 的 MergeTree 没有 UPDATE，重复写会产生重复行。

ReplacingMergeTree 的机制：指定一个 ver 列，在后台合并（merge）时，相同主键的行只保留 ver 最大的那条。这不是实时去重——在 merge 发生前，查询可能看到重复行，需要在查询时加 FINAL 关键字或用 argMax 聚合。

K 线表（ReplacingMergeTree） 
CREATE TABLE kline
(
    exchange    LowCardinality(String),
    symbol      LowCardinality(String),
    bar         LowCardinality(String),   -- '1m', '5m', '1h', '1d'
    open_time   DateTime64(3, 'UTC'),     -- 毫秒精度
    open        Float64,
    high        Float64,
    low         Float64,
    close       Float64,
    volume      Float64,
    quote_volume Float64,
    trades      UInt32,
    updated_at  DateTime64(3, 'UTC') DEFAULT now64()  -- 用作 ver
)
ENGINE = ReplacingMergeTree(updated_at)
PARTITION BY (exchange, toYYYYMM(open_time))
ORDER BY (exchange, symbol, bar, open_time)
TTL toDate(open_time) + INTERVAL 2 YEAR
SETTINGS index_granularity = 8192;

分区键：为什么按月而不是按天

常见的问题是分区粒度选太细——比如按天分区。ClickHouse 的分区数有上限建议（不超过数千个），每个分区也有元数据开销。

对于 K 线这类数据：

  • 按天：10 个交易所 × 2 年 = 7300 个分区，超过建议上限，写入时元数据操作变慢
  • 按月：10 个交易所 × 2 年 = 240 个分区，合理
  • 如果数据量很大（亿级），按月每个分区约 100–500MB，压缩后 30–100MB，ClickHouse 的列存压缩效果明显

另一个需要注意的是 PARTITION BY 里包含了 exchange——因为不同交易所的历史跨度和数据量差异大，不同交易所的数据落在不同分区，清理某个交易所的旧数据（ALTER TABLE DROP PARTITION）非常高效。

主键（ORDER BY）：查询路径决定字段顺序

ClickHouse 的 ORDER BY 决定数据在磁盘上的排序，也是稀疏索引的基础。字段顺序很重要：

  • 最左字段应该是选择性低的（cardinality 低），因为 ClickHouse 在扫描时利用 ORDER BY 做范围跳过
  • 常见查询是「某交易所某交易对某 bar 的时间范围」，所以顺序是 (exchange, symbol, bar, open_time)

注意 ORDER BY 也是 ReplacingMergeTree 的去重键——相同 (exchange, symbol, bar, open_time) 才会被视为同一条记录并去重。

批量写入：缓冲与异步

ClickHouse 对写入批次大小很敏感——每次写入会产生一个 part 文件，part 太小（几十行）会导致后台 merge 压力大，最终影响查询性能。官方建议每次写入不少于 1000 行。

异步批量写入（使用 clickhouse-connect） 
import asyncio
from clickhouse_connect import get_async_client
from collections import defaultdict

class ClickHouseBatchWriter:
    def __init__(self, host: str, flush_size: int = 5000, flush_interval: float = 5.0):
        self._host = host
        self._flush_size = flush_size
        self._flush_interval = flush_interval
        self._buffers: dict[str, list] = defaultdict(list)
        self._lock = asyncio.Lock()
        self._client = None

    async def connect(self):
        self._client = await get_async_client(host=self._host)

    async def insert(self, table: str, row: dict) -> None:
        async with self._lock:
            self._buffers[table].append(row)
            if len(self._buffers[table]) >= self._flush_size:
                await self._flush(table)

    async def _flush(self, table: str) -> None:
        rows = self._buffers.pop(table, [])
        if not rows:
            return
        await self._client.insert(table, rows, column_names=list(rows[0].keys()))

    async def flush_all(self) -> None:
        async with self._lock:
            for table in list(self._buffers.keys()):
                await self._flush(table)

TTL：自动清理旧数据

量化数据里有些类型不需要永久保留——比如 tick 级别的成交数据（占空间大，超过 3 个月基本不用回测），或者高频的 order book snapshot。ClickHouse 的 TTL 可以按行级别设置过期时间，过期后自动删除或转移到「冷」存储路径。

对 tick 数据设置 90 天 TTL 
CREATE TABLE trades_tick
(
    exchange    LowCardinality(String),
    symbol      LowCardinality(String),
    trade_time  DateTime64(3, 'UTC'),
    price       Float64,
    qty         Float64,
    side        Enum8('buy' = 1, 'sell' = -1),
    trade_id    String
)
ENGINE = ReplacingMergeTree
PARTITION BY (exchange, toYYYYMM(trade_time))
ORDER BY (exchange, symbol, trade_time, trade_id)
TTL toDate(trade_time) + INTERVAL 90 DAY
SETTINGS index_granularity = 8192;

典型查询模式

几个常见的量化查询写法：

查某交易对最近 30 天的日级 OHLCV（回测用） 
SELECT
    toDate(open_time) AS date,
    argMin(open, open_time)  AS open,
    max(high)                AS high,
    min(low)                 AS low,
    argMax(close, open_time) AS close,
    sum(volume)              AS volume
FROM kline FINAL
WHERE exchange = 'binance'
  AND symbol = 'BTCUSDT'
  AND bar = '1m'
  AND open_time >= now() - INTERVAL 30 DAY
GROUP BY date
ORDER BY date;
跨交易所资金费率对比（找套利窗口） 
SELECT
    exchange,
    symbol,
    funding_time,
    funding_rate,
    funding_rate * 3 * 365 AS annualized_rate   -- 8h 结算 × 3 × 365
FROM funding_rate FINAL
WHERE symbol = 'BTCUSDT'
  AND funding_time >= now() - INTERVAL 7 DAY
ORDER BY funding_time, exchange;

几个坑

  • FINAL 关键字的性能开销：SELECT ... FINAL 会触发实时合并，开销比普通查询高。如果数据写入频率低（比如每分钟一批），可以配置 deduplicate_blocks_in_dependent_materialized_views，或者定期手动 OPTIMIZE TABLE ... FINAL，然后在非 merge 窗口不用加 FINAL。
  • LowCardinality 类型：exchange、symbol、bar 这类低基数字符串，用 LowCardinality(String) 可以显著减少存储空间（内部用字典编码），查询时的过滤也更快。但如果某列的 distinct 值超过 10000，LowCardinality 反而会增加开销。
  • Float64 精度问题：价格用 Float64 存储时，对于低价代币（如 0.000001234）会有精度损失。可以考虑用 Decimal(18, 10)，但会增加存储空间和计算开销，需要根据场景权衡。
