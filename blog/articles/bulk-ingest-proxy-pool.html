系列：量化数据基础设施从零到生产 · #5

历史数据 10 亿行，45 分钟完成：代理池与并发设计

2025-12 · 约 10 分钟

产线上了线上实时采集之后，接下来要做的是把历史数据补全。KuCoin SPOT 的 1m K 线，最大可以追溯 4 年，约 220 个交易对，算下来大约 1.2 亿根 K 线，每次 API 请求最多拉 1500 根——需要 80000+ 次请求。

这篇记录从「跑三天」到「跑 45 分钟」的过程。

第一次跑：单线程单 IP，估计需要三天

最直接的实现：按交易对遍历，每个交易对按时间段分页拉取，逐一写入 S3。

KuCoin 的 rate limit 是 2000 weight / 30s，每次 candle 请求 weight=3，单 IP 上限约 22 req/s。80000 次请求 ÷ 22 req/s ≈ 60 分钟——但这是理论值，实际还要算上 HTTP 延迟、S3 写入时间、偶发的 429 退避。初步测试下来实际吞吐约 8–12 req/s，估计需要 2–3 天。

这个速度不可接受——每次新增一个交易所或数据类型，历史补数就要跑好几天。

瓶颈在哪？

单 IP 的问题是：rate limit 是绑定在 IP 上的。每个 IP 每 30 秒只能发 2000 weight。我们有单独的数据工程服务器，但所有请求都从同一个出口 IP 发出，所以无论怎么加线程、加进程，bottleneck 始终是这一个 IP 的配额。

解法直接：用多个 IP，每个 IP 各自有独立的配额。SOCKS5 代理池。

代理池设计

我们购买了一批 SOCKS5 代理（住宅 IP），每个代理有独立的出口 IP。代理池需要解决三个问题：

  1. 分配：把并发的拉取任务均匀分配到不同代理，避免某个代理过热
  2. 健康检查：代理会随机失效，要及时发现并隔离
  3. 故障转移：某个代理返回 429 时，换另一个代理重试，而不是等待
代理池（简化版） 
import asyncio
import aiohttp
from dataclasses import dataclass, field
from collections import deque

@dataclass
class Proxy:
    url: str           # "socks5://host:port"
    failures: int = 0
    in_use: bool = False

class ProxyPool:
    def __init__(self, proxy_urls: list[str], max_failures: int = 3):
        self._proxies = deque([Proxy(url=u) for u in proxy_urls])
        self._max_failures = max_failures
        self._lock = asyncio.Lock()

    async def acquire(self) -> Proxy | None:
        async with self._lock:
            for _ in range(len(self._proxies)):
                proxy = self._proxies[0]
                self._proxies.rotate(-1)
                if not proxy.in_use and proxy.failures < self._max_failures:
                    proxy.in_use = True
                    return proxy
        return None   # 所有代理都忙或故障

    async def release(self, proxy: Proxy, success: bool) -> None:
        async with self._lock:
            proxy.in_use = False
            if success:
                proxy.failures = 0
            else:
                proxy.failures += 1
                if proxy.failures >= self._max_failures:
                    # 把失效代理移到末尾等待人工检查
                    self._proxies.remove(proxy)
                    self._proxies.append(proxy)

    async def health_check(self, test_url: str = "https://api.kucoin.com/api/v1/timestamp"):
        """定期健康检查，重置已恢复的代理。"""
        async with aiohttp.ClientSession() as session:
            for proxy in list(self._proxies):
                try:
                    async with session.get(
                        test_url,
                        proxy=proxy.url,
                        timeout=aiohttp.ClientTimeout(total=5)
                    ) as resp:
                        if resp.status == 200:
                            proxy.failures = 0
                except Exception:
                    proxy.failures = min(proxy.failures + 1, self._max_failures)

每个代理独立限流

代理池解决了 IP 多样性，但每个 IP 自身仍然有 rate limit。从 上一篇 的经验来看，需要为每个代理维护独立的 Redis 限流 key，而不是共享一个全局 key。

每代理独立 Redis key 
from rate_limiter import RedisRateLimiter  # 上篇的实现

def make_proxy_limiter(proxy_url: str, redis_client) -> RedisRateLimiter:
    # 用代理 URL 的 hash 作为 Redis key 的命名空间
    proxy_key = hashlib.md5(proxy_url.encode()).hexdigest()[:8]
    return RedisRateLimiter(
        redis_client=redis_client,
        key=f"rate_limit:kucoin:{proxy_key}",
        limit=1800,    # 留 10% 余量（KuCoin 上限 2000 weight/30s）
        window=30,
    )

并发拉取流水线

任务的并发结构：把所有需要补全的（交易对, 时间段）对拆成独立任务，用 asyncio 并发执行，每个任务从代理池拿一个代理，拉完后归还。

并发拉取主循环 
async def fetch_one_chunk(
    symbol: str,
    start_ts: int,
    end_ts: int,
    proxy_pool: ProxyPool,
    proxy_limiters: dict,
    s3_client,
) -> None:
    proxy = None
    while proxy is None:
        proxy = await proxy_pool.acquire()
        if proxy is None:
            await asyncio.sleep(0.1)

    limiter = proxy_limiters[proxy.url]
    try:
        await limiter.acquire(weight=3)
        candles = await fetch_kucoin_candles(
            symbol=symbol,
            start_ts=start_ts,
            end_ts=end_ts,
            proxy=proxy.url,
        )
        if candles:
            await write_to_s3(candles, symbol, start_ts, s3_client)
        await proxy_pool.release(proxy, success=True)
    except Exception as e:
        await proxy_pool.release(proxy, success=False)
        raise

async def run_bulk_ingest(
    symbols: list[str],
    start_date: str,
    end_date: str,
    proxy_pool: ProxyPool,
    proxy_limiters: dict,
    s3_client,
    concurrency: int = 64,
):
    tasks = generate_tasks(symbols, start_date, end_date)  # → list[(symbol, start_ts, end_ts)]
    semaphore = asyncio.Semaphore(concurrency)

    async def bounded_fetch(task):
        async with semaphore:
            await fetch_one_chunk(*task, proxy_pool, proxy_limiters, s3_client)

    await asyncio.gather(*[bounded_fetch(t) for t in tasks], return_exceptions=True)

S3 中转，再批量写 ClickHouse

拉取结果先写 S3 Parquet，而不是直接插 ClickHouse，原因：

  • 断点续跑：某批代理挂了或网络中断，已经写进 S3 的部分不需要重拉；下次从 S3 中未入库的文件继续入库。
  • 批次写入效率：ClickHouse 对批量写入更友好，每次插入几十万行比逐条插入快几个数量级。
  • 原始数据留存：S3 的 raw 层保留原始 JSON/Parquet，入库逻辑有 bug 时可以重新 replay，不需要再打 API。

S3 按 exchange/data_type/year/month/ 分区存储，文件命名包含 symbol 和时间范围，方便检查哪些段已经完成。

S3 分区路径约定 
def s3_key(exchange: str, symbol: str, bar: str, start_ts: int) -> str:
    dt = datetime.utcfromtimestamp(start_ts / 1000)
    return (
        f"raw/{exchange}/kline/{bar}/"
        f"{dt.year}/{dt.month:02d}/"
        f"{symbol.replace('/', '_')}_{start_ts}.parquet"
    )

结果

32 个 SOCKS5 代理，每个代理约 22 req/s，理论峰值吞吐 32 × 22 = 704 req/s，实测约 710 req/s（有些代理延迟低，跑得快一些）。

KuCoin SPOT 4 年历史数据（约 220 个 symbol，1m K 线）：

方案 吞吐 耗时（估算）
单线程单 IP 8–12 req/s 2–3 天
32 代理 + 并发 ~710 req/s 约 45 分钟

S3 写入 → ClickHouse 批量导入约额外 30 分钟（ClickHouse 的 S3 table function 直接读取，不需要本地中转）。

踩过的坑

  • 代理质量差异大：30% 的代理会随机返回连接超时，要把 max_failures 调低，快速隔离差代理，而不是无限重试。
  • 同一代理的并发数不能太高：单个代理同时发 10+ 个请求，即使 weight 没超，也容易触发交易所的连接数限制（不同于 weight limit）。每代理并发数控制在 3–5 比较稳。
  • S3 小文件问题：每个时间段写一个文件，200 个 symbol × 时间分段会产生几万个小文件。入 ClickHouse 前先用 PyArrow 合并到合理大小（每文件 100MB 左右）。
  • 时区对齐：KuCoin 的时间戳是 UTC，但历史接口的 startAt/endAt 参数是秒级而不是毫秒级，和 Binance 的毫秒级接口混用时容易出 bug。
