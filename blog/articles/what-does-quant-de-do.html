量化公司的数据工程师在做什么？

2025-09 · 约 4 分钟 · 入门 · 量化

量化公司的数据工程师在做什么？不同背景的人理解差异很大。这篇文章梳理一下：量化公司需要哪些数据、数据从哪里来、以及数据工程在这里具体负责什么。

量化公司需要什么数据？

量化团队的核心工作是用数据和模型做交易决策。需要的数据大致分三类：

  • 市场行情：价格、成交量、K 线（按分钟/小时聚合）—— 来自各大交易所（Binance、OKX、Bybit 等）。这是最基础的原材料。
  • 衍生品特有数据：资金费率、持仓量（Open Interest）、多空比、强平数据等 —— 同样来自交易所，但接口和格式每家差异很大，需要分别处理。
  • 链上 / 另类数据：Polymarket 预测市场的成交记录、链上钱包行为、社交情绪指标（LunarCrush）、机构级标签数据（Nansen）等 —— 来源更杂，接入方式从 REST API 到 GraphQL 到直接读 RPC 都有。

这些数据有一个共同特点：来自很多个源头，格式不统一，每个源都有速率限制（超限会封 IP 或返回错误）。公司内部需要的是「随时能查到完整、及时、一致的数据」——把「多源、受限、杂乱」变成「统一、稳定、可查」，就是数据工程师的核心任务。

数据工程师具体做什么？

可以拆成四块：

  1. 接进来：把多家交易所、多种数据类型的 API 抽象成统一的接口层（Endpoint / Reader），写拉取逻辑、限流、重试。新增一个交易所或一种数据类型时，不用重写整套流程。
  2. 存好：设计表结构、分区键、去重规则，让数据能按时间、按交易所、按交易对高效查询。历史数据一般先落 S3（Parquet），再批量导入 ClickHouse；与实时流共用同一套 schema 和幂等写入逻辑。
  3. 不丢：用调度系统（Prefect）定时跑拉取任务，失败自动重试；监控任务状态和日志，发现数据缺口立即告警或触发补数。
  4. 能维护：交易对上下架频繁，需要一套符号生命周期管理——从交易所同步最新列表，用 YAML + Git 管理变更，Slack 通知，下架走人工审批，避免误删。

为什么量化的数据工程比想象中难？

表面上看，「定时调 API，写数据库」听起来没什么难度。实际的复杂度来自几个地方：

  • 交易所的 API 行为差异很大——分页方向不统一、限流粒度不一、历史数据接口和实时接口分离，而且文档经常不准确，要靠实测。
  • 多进程环境下的限流是个系统性问题：每个进程各自计数会导致实际超配额，需要全局共享的原子计数器（Redis 滑动窗口那篇讲了具体案例）。
  • 历史数据的体量和接入链路不一样——10 亿行数据用单线程单 IP 拉是不现实的，需要代理池 + 并发任务协同（这在系列后续文章里展开）。
  • 链上数据的工程模式和 CEX API 完全不同，The Graph 的分页机制、subgraph 的延迟管理都是独立的知识体系。
