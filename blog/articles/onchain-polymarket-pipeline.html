链上数据接入：Polymarket 从 subgraph 到 ClickHouse 的完整流程

2025-12 · 约 11 分钟

链上数据的接入和 CEX API 有本质不同——没有统一的 REST 接口，数据散落在链上日志里，需要通过索引层（如 The Graph）或直接读 RPC 才能拿到结构化数据。Polymarket 运行在 Arbitrum 上，The Graph 为它维护了专门的 subgraph，暴露了市场、交易、流动性等数据。

这篇记录完整的接入流程：从 subgraph 查询到增量同步设计，到 S3 暂存，再到 ClickHouse 入库和数据质量检验。

Polymarket 的数据模型

Polymarket 的预测市场每个问题有一个 Market，每个 Market 分两个 Outcome（Yes/No），用户可以买卖对应 Outcome 的份额。数据模型里几个核心实体：

  • Market：问题描述、结局、创建时间、成交量、结算结果
  • Trade：每一笔成交——时间戳、市场、Outcome、方向（buy/sell）、数量、价格
  • ConditionId：Polymarket 用来唯一标识 Market 的链上 ID（与 Gnosis Conditional Tokens 协议绑定）

我们主要关心 Trade——类似 CEX 的 tick 数据，但来自链上事件日志而不是 REST API。

Subgraph 查询基础

Polymarket 的官方 subgraph 部署在 The Graph Network 上，GraphQL 端点大致是：

https://gateway.thegraph.com/api/{API_KEY}/subgraphs/id/{SUBGRAPH_ID}

查询 Trade 的基本结构：

查询最近的成交记录 
query GetTrades($market: String!, $first: Int!, $lastId: ID!) {
  trades(
    first: $first
    where: {
      market: $market
      id_gt: $lastId
    }
    orderBy: id
    orderDirection: asc
  ) {
    id
    timestamp
    market { id, question }
    outcomeIndex
    side          # BUY / SELL
    size          # 份额数量（USDC）
    price         # 价格（0–1 之间）
    transactionHash
  }
}

注意用 id_gt 而不是 skip 翻页——The Graph 的 skip 有 5000 上限，超过后返回空结果，且不报错。这个坑在数据全景那篇有解释。

全量初始化：~10M 条记录

第一次接入需要把所有历史 Trade 都拉下来。Polymarket 的 Trade 总量约 10M 条，每次最多拉 1000 条，需要约 10000 次请求。

全量分页拉取（cursor-based） 
import asyncio
import aiohttp
import pyarrow as pa
import pyarrow.parquet as pq
from pathlib import Path

QUERY = """
query GetAllTrades($lastId: ID!, $first: Int!) {
  trades(
    first: $first
    where: { id_gt: $lastId }
    orderBy: id
    orderDirection: asc
  ) {
    id
    timestamp
    market { id }
    outcomeIndex
    side
    size
    price
    transactionHash
  }
}
"""

SCHEMA = pa.schema([
    pa.field("id", pa.string()),
    pa.field("timestamp", pa.int64()),
    pa.field("market_id", pa.string()),
    pa.field("outcome_index", pa.int8()),
    pa.field("side", pa.string()),
    pa.field("size", pa.float64()),
    pa.field("price", pa.float64()),
    pa.field("tx_hash", pa.string()),
])

async def fetch_page(session: aiohttp.ClientSession, url: str, last_id: str) -> list[dict]:
    async with session.post(url, json={
        "query": QUERY,
        "variables": {"lastId": last_id, "first": 1000}
    }) as resp:
        data = await resp.json()
        return data["data"]["trades"]

async def full_backfill(subgraph_url: str, output_dir: Path):
    output_dir.mkdir(parents=True, exist_ok=True)
    last_id = ""
    page = 0

    async with aiohttp.ClientSession() as session:
        while True:
            rows = await fetch_page(session, subgraph_url, last_id)
            if not rows:
                break

            # 转成 Parquet 写入 S3（这里写本地，实际用 s3fs）
            records = [{
                "id": r["id"],
                "timestamp": int(r["timestamp"]),
                "market_id": r["market"]["id"],
                "outcome_index": int(r["outcomeIndex"]),
                "side": r["side"],
                "size": float(r["size"]),
                "price": float(r["price"]),
                "tx_hash": r["transactionHash"],
            } for r in rows]

            table = pa.Table.from_pylist(records, schema=SCHEMA)
            pq.write_table(table, output_dir / f"trades_page_{page:06d}.parquet")

            last_id = rows[-1]["id"]
            page += 1

            # 友好地控制请求频率
            await asyncio.sleep(0.05)

增量同步：checkpoint 机制

全量初始化完成后，每小时增量拉取新成交。增量同步的核心是记录上次同步的位置——用 ClickHouse 里最新的一条记录的 id 作为 checkpoint，下次从这个 id 之后继续。

增量同步 Prefect flow 
from prefect import flow, task
import clickhouse_connect

@task
def get_last_synced_id(ch_client) -> str:
    """从 ClickHouse 查上次同步的最新 trade id。"""
    result = ch_client.query(
        "SELECT max(id) FROM polymarket_trades"
    )
    last_id = result.first_row[0]
    return last_id or ""

@task
async def fetch_new_trades(subgraph_url: str, last_id: str) -> list[dict]:
    """拉取 last_id 之后的所有新成交。"""
    all_trades = []
    async with aiohttp.ClientSession() as session:
        cursor = last_id
        while True:
            rows = await fetch_page(session, subgraph_url, cursor)
            if not rows:
                break
            all_trades.extend(rows)
            cursor = rows[-1]["id"]
            await asyncio.sleep(0.05)
    return all_trades

@task
def write_to_clickhouse(trades: list[dict], ch_client):
    if not trades:
        return
    ch_client.insert("polymarket_trades", trades,
                     column_names=["id", "timestamp", "market_id",
                                   "outcome_index", "side", "size",
                                   "price", "tx_hash"])

@flow
def polymarket_incremental_sync():
    ch_client = clickhouse_connect.get_client(host="localhost")
    last_id = get_last_synced_id(ch_client)
    new_trades = fetch_new_trades.fn(SUBGRAPH_URL, last_id)  # 异步调用
    write_to_clickhouse(new_trades, ch_client)

ClickHouse 表设计

Trade 数据按天分区，以 id（链上 log 的唯一标识）作为去重键：

polymarket_trades 表结构 
CREATE TABLE polymarket_trades
(
    id              String,
    trade_time      DateTime64(0, 'UTC'),
    market_id       String,
    outcome_index   Int8,        -- 0: No, 1: Yes
    side            Enum8('BUY' = 1, 'SELL' = -1),
    size            Float64,     -- USDC 金额
    price           Float32,     -- 0.0 ~ 1.0（隐含概率）
    tx_hash         String,
    ingested_at     DateTime DEFAULT now()
)
ENGINE = ReplacingMergeTree
PARTITION BY toYYYYMMDD(trade_time)
ORDER BY (market_id, trade_time, id)
TTL toDate(trade_time) + INTERVAL 3 YEAR;

数据质量检验

链上数据的质量问题和 CEX API 不一样，需要针对性地检验：

Subgraph 延迟检测

每次同步前检查 subgraph 的当前区块高度与链上最新区块的差距（见数据全景那篇的代码）。如果 lag > 50 块（Arbitrum 约 12 秒），记录告警但不中断同步——lag 大时数据是滞后的，但不是错的。

价格边界检验

Polymarket 的 price 应该在 (0, 1) 之间（除了最后结算的瞬间）。写入前过滤掉 price = 0 或 price = 1 的记录——这通常是结算事件，而不是正常成交，混入会污染回测数据。

写入前的简单校验 
def validate_trade(trade: dict) -> bool:
    price = float(trade["price"])
    size = float(trade["size"])
    return (
        0 < price < 1          # 不含边界值
        and size > 0
        and trade["side"] in ("BUY", "SELL")
        and trade["tx_hash"].startswith("0x")
    )

连续性检查

每天生成一张数据质量报告：按市场统计当天成交笔数，与前一天对比。成交量骤降（>90%）可能意味着市场已结算（正常）或 subgraph 同步中断（异常）——需要结合市场的结算状态一起判断。

日级成交量环比检查（ClickHouse SQL） 
WITH daily AS (
    SELECT
        toDate(trade_time) AS dt,
        market_id,
        count() AS trades_count,
        sum(size) AS total_volume
    FROM polymarket_trades FINAL
    WHERE dt >= today() - 7
    GROUP BY dt, market_id
)
SELECT
    today_data.market_id,
    today_data.trades_count    AS today_trades,
    yesterday_data.trades_count AS yesterday_trades,
    today_data.trades_count / greatest(yesterday_data.trades_count, 1) AS ratio
FROM daily today_data
LEFT JOIN daily yesterday_data
    ON today_data.market_id = yesterday_data.market_id
    AND yesterday_data.dt = today_data.dt - 1
WHERE today_data.dt = today()
  AND ratio < 0.1   -- 成交量较前一天下降 90% 以上
ORDER BY ratio;

几个坑

  • The Graph API key 的速率限制：免费 tier 的 API key 每分钟请求数有限，全量初始化时要控制节奏（sleep 50ms 每请求已经接近上限）。如果是长期生产使用，建议用付费 tier 或自建 subgraph 节点。
  • Trade id 不是纯递增的数字：Polymarket subgraph 里 Trade 的 id 是 {txHash}-{logIndex} 格式的字符串，字典序和时间序大致一致，但不完全一致（同一区块内不同 log 的 id 字典序可能与 logIndex 顺序不符）。checkpoint 用 id 是安全的（不会漏数据），但查询时要用 timestamp 而不是 id 做时间范围过滤。
  • 市场结算后的数据：市场结算后，The Graph 里该 market 的 resolved 字段变为 true，价格会固定在 0 或 1。增量同步时不需要特殊处理，但如果分析时要过滤掉结算后的「交易」，需要 join market 表查 resolved_at。
  • 重复 trade：The Graph 的 subgraph 在极少数情况下（节点重组时）会重复返回同一笔 trade。ClickHouse 的 ReplacingMergeTree 加上 FINAL 查询会处理重复，但全量初始化时的 Parquet 文件里可能有重复行，入库前最好做一次去重。
