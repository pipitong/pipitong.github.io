系列：量化数据基础设施从零到生产 · #2

Crypto 量化数据全景：策略、数据分类与工程笔记

2025-10 · 约 15 分钟

理解「需要什么数据」之前，要先理解「在跑什么策略」。这篇从策略出发，把 crypto 量化涉及的数据系统梳理一遍——六大类，30+ 种具体数据类型——同时记录每类数据在工程接入时的细节和坑。

一、Crypto Quant 在做什么

策略决定数据需求。不同策略对数据的新鲜度、覆盖范围、历史深度要求完全不同：

策略类型 核心逻辑 核心数据需求
Market Making 挂双边报价，赚 spread，控 inventory risk Tick、L2/L3 order book、funding rate（持仓成本）
Funding Rate Arb 做多/空永续，赚资金费率，现货对冲敞口 多交易所 funding rate、spot 价格、borrow rate
Cross-exchange Arb 同一币种在不同交易所的瞬时价差 多交易所 trade/order book、延迟监控
CTA / Trend Following 动量、均值回归等系统化策略 OHLCV、成交量、OI、funding rate
Options / Vol Trading 波动率曲面交易、delta-neutral 对冲 IV surface、options chain、历史 realized vol
On-chain / DeFi MEV、LP 流动性提供、链上套利 DEX swaps、mempool、链上状态
Prediction Market 对事件概率的定价 Polymarket 链上成交记录、流动性深度
Alt-data Driven 社交情绪、聪明钱行为预测价格 LunarCrush、Nansen whale 钱包追踪

二、数据分类全景

1. 市场数据（CEX）

OHLCV / K 线 已接入

来源：Binance、OKX、Bybit、Coinbase、KuCoin、Bitget、Gate、Hyperliquid — 历史 + 实时

粒度：1m / 5m / 15m / 1h / 4h / 1d，各交易所均支持

字段：open_time, open, high, low, close, volume, quote_volume, trades

工程细节：OKX /market/candles 只返回最近 1440 根，超过需切到 /market/history-candles（不同接口路径、不同频率限制）。各交易所分页方向不统一——Binance 升序，OKX 降序，KuCoin Futures 升序而 KuCoin Spot 降序。时间戳单位：Binance/OKX 是毫秒，KuCoin 是秒。

Trade Data（逐笔成交）

来源：各交易所公开 API 提供最近 N 笔；历史 tick 数据用 Databento（专业来源，按月购买，MBO/MBP 格式，主要覆盖 CME 和部分 CEX）

字段：trade_time, price, qty, side (taker buy/sell), trade_id

用途：做市策略的核心输入；计算 taker buy/sell ratio（短期方向指标）；构建 VWAP

工程细节：实时 tick 数据体量很大（BTC/USDT 高峰期每秒数千笔），不宜无差别存储。常见做法是只存聚合统计（每分钟 taker volume 比）或按需存原始 tick 并设短 TTL（90 天）。

Order Book（L1 / L2 / L3）

L1：Best bid / best ask（最优报价），所有交易所均提供

L2：价格档位 + 各档位总量（depth），通常返回 top 20/50/100 档

L3：带 order ID 的逐笔挂单（订单级别），只有少数交易所提供（Coinbase Advanced、BTSE 等）

用途：做市策略实时感知市场深度；套利策略判断可执行价差

工程细节：Order book snapshot 体量极大，通常走 WebSocket 增量更新而不是 REST 轮询。本地维护 order book 状态需要处理 sequence number 校验和断连重建，是独立的工程问题。

Ticker / Mark Price / Index Price

Ticker：实时行情快照（最新成交价、24h 涨跌、成交量）

Mark Price：交易所合成的「公允价格」，用于计算未实现盈亏和强平价格，比最新成交价更稳定（加权多个来源）

Index Price：多交易所加权合成的指数价，防止单一交易所价格被操纵（特别重要于期权定价）

工程细节：Mark Price 和 Index Price 很多交易所走 WebSocket 推送，频率高（每秒多次），REST 轮询可能跟不上。对大多数非 HFT 策略，1 分钟 REST 快照已够。

2. 衍生品数据（Perps & Futures）

Funding Rate（资金费率）已接入

来源：Binance、OKX、Bybit、KuCoin、Bitget、Gate、Hyperliquid

结算频率：Binance/OKX 每 8 小时（00:00 / 08:00 / 16:00 UTC），Bybit/Hyperliquid 每小时

字段：symbol, funding_time, funding_rate, (predicted_rate)

工程细节：跨交易所对比时要先年化（rate × 结算次数/天 × 365），否则 Binance 的 8h 费率和 Bybit 的 1h 费率直接相减没有意义。「预测资金费率」和「已结算资金费率」来自不同接口——在结算前，交易所会实时计算当前期的预测值，一些策略需要这个而不是历史结算值。

Open Interest（持仓量，OI）已接入

来源：各 CEX 的 futures stats 接口

字段：symbol, timestamp, open_interest_value（USDT 计）, open_interest_amount（币本位）

用途：OI 上升 + 价格上升 → 多方入场；OI 骤降 → 大量平仓（强平或主动）；OI 变化量有时比绝对值更有信号

工程细节：注意 OI 的计价单位——同一个合约的 OI 可以用 USD、USDT、BTC 三种单位表示，跨交易所汇总时要统一到同一单位再加总。

Long/Short Ratio（多空比）已接入

三个口径，含义不同，不能混用：

  • 全市场账户多空比（Account L/S Ratio）：所有用户中，净多账户 vs 净空账户数量。散户情绪代理，噪声大。
  • 顶级交易者仓位多空比（Top Trader Position Ratio）：头部用户的多头仓位量 vs 空头仓位量。大资金方向，信号质量更高。
  • 顶级交易者账户多空比（Top Trader Account Ratio）：头部用户中，净多账户 vs 净空账户数量。与上一个是不同统计维度。

另有 Taker Buy/Sell Volume Ratio（主动买卖量比），反映短期市场主动方向，是第四个独立维度。

工程细节：这四个指标要存在不同字段（或不同表），写入时加 ratio_type 枚举列区分。混在一起后期很难清理。OKX 提供全部四种口径，Binance 只提供前三种。

Liquidations（强平）

来源：Binance WebSocket forceOrder stream、OKX WebSocket；历史数据较难拿全

字段：symbol, side, price, qty, trade_time

用途：爆仓瀑布预警——大量空头被强平往往伴随价格快速上涨，反之亦然；强平额骤升是短期波动率飙升的先行指标

工程细节：Binance 的 liquidation WebSocket 只推送被强平的订单，不是实时完整的强平量统计。若要跨交易所汇总强平数据，Coinglass 等第三方平台有聚合 API。

Basis（基差）

定义：现货价格 vs 期货/永续合约价格的差值（永续合约通常称 premium），反映期限溢价和市场情绪

计算方式：basis = (perp_price − spot_price) / spot_price × 100%

用途：资金费套利的核心输入；高正基差通常对应高资金费率（多头付费给空头）

工程细节：Basis 通常不需要专门接口，实时计算即可——从 spot 和 perp 的 mark price 做差。重点是确保两个来源的时间戳对齐，否则算出来的 basis 包含时间错位的噪声。

3. 期权数据

Implied Volatility（隐含波动率）已接入（OKX priAPI）

来源：OKX priAPI（路径 priapi/v5，非官方公开文档）、Deribit API

关键指标：

  • ATM IV：平值期权的隐含波动率，「市场定价」的波动率基准
  • IV Skew：虚值看跌期权 IV 与虚值看涨期权 IV 的差，反映市场对下跌风险的定价
  • IV vs Historical Realized Vol：隐波 vs 历史实波的差（vol premium），是均值回归类波动率策略的核心信号
  • Term Structure：不同到期日的 ATM IV 曲线，用于跨期日的套利和对冲
工程细节：OKX 公开 API（api/v5）提供的期权数据很有限，机构关注的 IV surface、skew 等指标在 priAPI 里，该接口无官方文档，需要抓包获取参数格式，且可能随时变更——要加 schema 版本记录和告警。Deribit 是 crypto 期权流动性最深的交易所，IV 数据通常以 Deribit 为基准。

Options Chain

来源：Deribit（主要）、OKX

字段：strike, expiry, bid/ask, last_price, open_interest, delta, gamma, vega, theta

用途：期权做市需要全链数据；delta-neutral 对冲需要实时 Greeks；构建 IV surface 需要所有 strike 的 IV 截面

工程细节：期权链的 symbol 数量随到期日临近快速变化（每周到期导致 symbol 频繁新增/下架）；strike 之间间距随 BTC 价格区间动态调整。标准化存储需要把 strike/expiry 从 symbol 字符串里解析出来单独存列。

Put/Call Ratio（PCR）已接入

两种口径：

  • OI PCR：看跌期权未平仓量 / 看涨期权未平仓量，反映市场结构性偏向
  • Volume PCR：当日成交量口径，反映短期方向情绪
工程细节：PCR 可以直接从 options chain 聚合计算，不需要单独接口——但要注意按到期日分组计算近月 PCR 和远月 PCR，两者含义不同。

4. 链上数据（On-chain）

原始交易 / 事件

DEX Swaps：Uniswap / Curve / Balancer 等 DEX 的 swap 事件，含 token_in, token_out, amount, sender, recipient

ERC-20 Transfer events：所有代币的转账记录，可追踪大额资金流动

接入方式：直接读节点 RPC（eth_getLogs）或使用 The Graph subgraph

工程细节：直接读 RPC 延迟最低（秒级），但需要自己 decode ABI，历史查询慢且节点成本高。The Graph 延迟约 1–5 分钟（Arbitrum 区块时间 ~0.25s，subgraph 通常落后几十个区块），适合历史数据和分钟级更新场景。

Nansen Smart Money 已评估

内容：500+ 标记钱包（机构、DEX 套利者、早期投资者等）的历史链上行为，覆盖 3 年以上

用途：追踪「聪明钱」的仓位变动；提前发现某个代币的机构积累行为

工程细节：API 按查询量计费，评估阶段要先做样本测试；钱包标签是 Nansen 自己的私有模型，有黑盒成分；主要覆盖以太坊主网，Arbitrum 和 Solana 的覆盖深度较浅。

Polymarket 链上成交 已接入，~10M records

来源：The Graph（Arbitrum subgraph）

字段：market_id, outcome_index, side (BUY/SELL), size (USDC), price (0–1), transaction_hash

特点：price 本身是隐含概率，直接反映市场对事件发生概率的定价

工程细节：The Graph 的 skip 参数上限 5000，必须用 cursor-based pagination（id_gt）翻页。全量 10M 条初始化约需 10000 次请求；增量同步用 ClickHouse 中最新 trade ID 作 checkpoint。Trade ID 格式是 {txHash}-{logIndex}，字典序和时间序大致一致但不完全一致，范围查询要用 timestamp 字段。

Exchange Flows（交易所净流入/流出）

内容：大额资金从链上钱包转入/转出 CEX 的量，是 CEX 卖压的早期信号（转入交易所 → 可能卖出）

来源：CryptoQuant、Glassnode、Nansen

工程细节：这类数据需要识别交易所的冷热钱包地址，各家数据商的标记方法不同，同一笔转账在不同数据源可能被归类为「交易所流入」或「内部转账」。

DeFi 协议数据

TVL（锁仓量）：各协议的总锁仓资产价值，来自 DeFiLlama API（免费，覆盖最全）

Lending Rates：Aave / Compound 的借贷利率，影响链上杠杆成本，和 CEX funding rate 存在套利关系

LP 池组成：DEX 流动性池的 token 比例和手续费收益，用于 LP 策略

链上状态数据

  • Active Addresses：每日活跃地址数，链上活跃度代理，来自 Glassnode
  • Gas Price / Mempool：实时 gas 价格和 pending 交易量，MEV 策略的核心输入
  • Stablecoin Mint/Burn：USDT / USDC 的增发和销毁，大额增发通常先于资金流入 CEX

5. 跨市场 / 宏观数据

CME 比特币期货 已接入

来源：Databento（历史 tick，MBO/MBP 格式，按月购买）

用途：机构持仓分析（CME 的 OI 以机构资金为主）；CME 期货 vs 永续合约的 basis 套利；CoT（Commitment of Traders）报告追踪机构净多空

工程细节：Databento 的数据格式是 MBO（Market By Order，逐订单级别），需要专门的 schema 解析，和 CEX API 的 JSON 完全不同，不能用同一套 reader 处理。历史数据按月购买，文件大（数 GB），通常先落 S3 再解析入库。

XNAS / 股票价格 已接入，post-adjusted

内容：纳斯达克上市公司的股价（主要关注 NASDAQ 指数和 COIN/MSTR 等 crypto 相关标的）

用途：BTC vs NASDAQ 的相关性研究；risk-off 环境下的宏观择时

注：post-adjusted 价格已做拆股/并股调整，适合长期历史分析

DXY / 美联储利率 / 国债收益率

DXY（美元指数）：强势美元通常对应 BTC 承压（risk-off），是宏观择时的重要信号

美联储利率 / SOFR：无风险利率水平影响加密市场的整体风险偏好

10Y 美债收益率：与 BTC 的相关性在 2022–2023 年显著，加息预期上升 → BTC 通常下跌

来源：FRED API（美联储经济数据，免费）、Quandl/Nasdaq Data Link

6. 另类 / 社交数据（Alt-data）

LunarCrush 已接入，~80M 数据点

内容：3000+ 资产的社交情绪数据（小时粒度），覆盖 3 年以上历史；指标包括 Galaxy Score、AltRank、social_volume、social_engagement

工程细节：注意幸存者偏差——API 返回的是当前仍存在的资产，已下架小币种的历史数据不完整。社交量有明显的尖峰噪声（bot 刷量、媒体爆炸性报道），直接用原始值做信号需要做 robust normalization。LunarCrush 偶尔会回填历史数据，增量拉取要设计好 upsert 逻辑。

Nansen Smart Money 已评估

内容：机构级别的链上钱包标签和行为追踪

（见上文链上数据部分）

Glassnode

内容：BTC/ETH 的链上宏观指标，包括 MVRV（市值 vs 已实现市值）、SOPR（已实现利润率）、NVT（网络价值 vs 交易量）等

用途：判断市场估值高低的宏观工具，更多用于研究和因子构建，不适合高频信号

Santiment / 新闻事件

Santiment：社交媒体情绪 + 链上开发活动（GitHub commits）+ 鲸鱼交易，覆盖 1500+ 资产

新闻事件：监管动态（SEC 诉讼、ETF 审批）、项目进展（升级、黑客攻击）—— 对短期价格冲击影响大，难以系统化，通常作为事件驱动策略的输入

Fear & Greed Index

来源：Alternative.me API（免费）

内容：0（极度恐惧）到 100（极度贪婪）的综合情绪指数，输入包括波动率、成交量、社交媒体、市值占比等

注：这个指标更像是市场情绪的摘要，而不是原始信号。单独用 Fear & Greed 做策略的回测结论通常过拟合。

三、数据新鲜度 vs 用途

不同数据类型对延迟的要求差异是量级级别的，决定了接入方式的选择：

延迟要求 数据类型 接入方式 典型策略
< 1ms L2/L3 order book、逐笔 trade WebSocket 直连，co-location 做市、高频套利
< 1s Mark price、liquidation stream WebSocket 套利、风控触发
< 1min OHLCV、OI、funding rate、taker volume REST 定时拉 本系统 CTA、信号生成、实时监控
< 1h IV surface、社交指标、链上聚合 REST 定时拉 中频策略、波动率交易
< 1d CME CoT、宏观数据、DeFi TVL、Glassnode REST 日级拉取 因子研究、宏观择时
历史 所有数据类型的历史补全 批量入湖（代理池 + 并发）本系统 回测、因子挖掘

四、工程视角的选型原则

面对一个新数据源，工程上先想清楚三件事：

  1. 新鲜度要求：分钟级 REST 够还是需要 WebSocket；链上数据接受 1–5 分钟 subgraph 延迟还是需要直接读 RPC
  2. 历史深度：交易所 API 通常只保留最近几个月到一年，更早的历史需要找专业归档来源（Databento）或接受数据不完整
  3. 质量校验策略：K 线用时间戳连续性和成交量异常检测；资金费率用结算时间对齐；链上数据用 subgraph lag 监控和价格边界检验；alt-data 警惕幸存者偏差和历史回填
真正的难点不是「能不能拿到数据」，而是「知道数据什么时候是错的」。
