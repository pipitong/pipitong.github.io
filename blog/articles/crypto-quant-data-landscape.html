系列：量化数据基础设施从零到生产 · #2

Crypto 量化数据全景：从市场行情到链上的工程笔记

2025-10 · 约 12 分钟

量化公司要用的数据远不止 K 线。从交易所行情、衍生品指标、期权波动率，到链上交易记录和社交情绪，每一类数据背后都有独立的接入方式和工程细节。这篇文章梳理一下这张地图，以及每类数据在实际接入时遇到的真实问题。

数据全景

大致可以分为六类：

分类 典型数据 来源 接入方式
CEX 市场数据 OHLCV、逐笔成交、Order Book Binance、OKX、Bybit 等 REST / WebSocket
衍生品数据 资金费率、持仓量、多空比、强平 各 CEX 统计接口 REST（定时拉）
期权数据 IV、skew、PCR、term structure OKX priAPI、Deribit REST / WebSocket
链上数据 DEX 交易、钱包追踪、预测市场 The Graph、RPC、Nansen GraphQL / JSON-RPC
跨市场 / 宏观 CME 期货、股票价格、美联储利率 CME、XNAS、Databento REST / FTP
另类数据 社交情绪、钱包标签、新闻事件 LunarCrush、Nansen、Santiment REST

下面按类逐一说工程细节。

K 线（OHLCV）：看起来最简单，坑最多

K 线是最基础的数据，也是最容易踩坑的地方，因为各交易所的接口行为差异很大。

历史窗口限制

大多数交易所的 K 线接口只返回最近 N 根（Binance 是 1000，OKX 是 1440）。要拉更早的历史数据，要么分页往前翻，要么走专门的「历史归档」接口——但两者的 API 路径不同，返回格式也可能有差异。

OKX 的情况尤其典型：/market/candles 只能拿最近 1440 根，超过这个范围要切到 /market/history-candles。这两个接口的参数名相同，但 history-candles 对频率限制更严，且不支持实时数据，只能查历史。拉历史 K 线的代码如果不做自动路由，很容易在 1440 根处截断而不报错。

OKX 历史 K 线自动路由（简化版） 
REALTIME_MAX_BARS = 1440  # /market/candles 的上限

def fetch_okx_candles(inst_id: str, bar: str, start_ts: int, end_ts: int) -> list:
    """
    自动判断是否需要走 history-candles 接口。
    start_ts, end_ts 均为毫秒时间戳。
    """
    now_ms = int(time.time() * 1000)
    bar_ms = bar_to_milliseconds(bar)  # "1m" → 60_000

    # 如果请求范围超出 1440 根窗口，走 history-candles
    cutoff_ms = now_ms - REALTIME_MAX_BARS * bar_ms
    if start_ts < cutoff_ms:
        endpoint = "/api/v5/market/history-candles"
    else:
        endpoint = "/api/v5/market/candles"

    return _paginate_candles(endpoint, inst_id, bar, start_ts, end_ts)

分页方向不统一

Binance 的 K 线接口默认按时间升序返回；OKX 默认降序；KuCoin Spot 接口降序，而 KuCoin Futures 接口升序——同一家交易所的两个产品线行为相反。批量拉历史数据时，分页逻辑要单独处理，否则会漏数据或重复请求。

时间戳：左闭右开还是右闭

一根 1 分钟 K 线的 open_time 是这根 K 线的开始时间（左闭），这是共识。但在用 startTime/endTime 做范围查询时，各家对边界的处理不完全一样。最稳妥的做法是请求时多留半根 K 线的缓冲，写入 ClickHouse 时用 ReplacingMergeTree 去重。

资金费率：细节决定信号质量

资金费率（Funding Rate）是永续合约特有的机制，用于锚定合约价格和现货价格。对量化策略来说，它既是成本（持仓要付/收资金费），也是信号（费率高低反映市场偏向）。

结算时间不对齐

不同交易所的资金费率结算频率和时间点不同：

交易所 频率 结算时间点（UTC）
Binance 每 8 小时 00:00 / 08:00 / 16:00
OKX 每 8 小时 00:00 / 08:00 / 16:00
Bybit 每小时 每整点
Hyperliquid 每小时 每整点

做跨交易所对比时需要注意：Binance 8h 一条记录，Bybit 1h 一条记录，直接对齐会算出假相关。正确做法是把所有交易所的 funding rate 统一归一化为 annualized rate（年化）再比较。

已结算 vs 预测资金费率

大多数接口返回的是「已结算」的历史资金费率。但有些策略需要的是「下一期预测资金费率」——在结算前的这段时间，交易所会实时计算当前 funding rate（如果现在结算会是多少），可以从 ticker 或专门的 premium index 接口获取。两者混用会引入信号滞后。

多空比：三档的区别

「多空比」容易引起混淆，因为实际上有三种口径，含义不同：

  • 全市场账户多空比（Long/Short Account Ratio）：所有用户中，净多仓账户 vs 净空仓账户的比例。反映散户情绪，噪声大。
  • 顶级交易者持仓多空比（Top Trader Position Ratio）：按持仓规模排名靠前的交易者，其多头持仓 vs 空头持仓的仓位比。反映大资金方向，信号质量更高。
  • 顶级交易者账户多空比（Top Trader Account Ratio）：顶级交易者中，净多账户 vs 净空账户的数量比。和上一个是不同的统计方式。

OKX 还单独提供了 Taker Buy/Sell Volume（主动买卖量比），这是另一个维度——反映的是短期的市场主动方向，而不是仓位方向。

这几个指标在数据库里一定要用不同的表或字段区分，混在一起后期很难清理。

期权数据：不在公开 API 里的内容

以 OKX 为例，公开 API（api/v5）里的期权数据覆盖很有限，很多机构关注的指标——ATM IV、IV skew、implied vs historical vol——只在 priAPI（私有 API，路径 priapi/v5） 里提供，这个接口没有官方文档，需要通过抓包或第三方整理的方式发现。

这类接口的不稳定性比公开 API 高，参数格式可能随时变，最好做好 schema 版本记录和告警。

链上数据：完全不同的工程模式

链上数据的接入方式和 CEX REST API 有本质不同，工程模式需要重新设计。

三种接入方式的比较

方式 延迟 适合场景 主要缺点
The Graph (subgraph) 1–5 分钟 behind 历史数据批量、协议聚合数据 subgraph 自定义查询受限；网络拥塞时延迟更大
直接 RPC（eth_getLogs 等） 秒级 低延迟监控、自定义过滤 需要自己 decode ABI；节点成本高；历史查询慢
第三方 API（Nansen、Dune 等） 分钟到小时 标签化数据、聚合分析 成本高；API 覆盖范围有限；不可控

The Graph：分页的坑

The Graph 的 GraphQL 接口支持 first 和 skip 分页，但 skip 有上限（通常是 5000），超过后返回空结果而不报错。正确的方式是用 cursor-based pagination：以上一批最后一条记录的 ID 作为下一次查询的起始点。

The Graph cursor 分页（避免 skip 上限） 
QUERY = """
query GetTrades($lastId: ID!, $first: Int!) {
  trades(
    first: $first
    where: { id_gt: $lastId }
    orderBy: id
    orderDirection: asc
  ) {
    id
    timestamp
    market
    outcome
    amount
    price
  }
}
"""

def fetch_all_trades(subgraph_url: str, batch_size: int = 1000) -> list:
    last_id = ""   # 空字符串在 GraphQL ID 比较中小于所有实际 ID
    results = []

    while True:
        resp = requests.post(subgraph_url, json={
            "query": QUERY,
            "variables": {"lastId": last_id, "first": batch_size},
        })
        batch = resp.json()["data"]["trades"]
        if not batch:
            break

        results.extend(batch)
        last_id = batch[-1]["id"]   # 下一页从这里开始

    return results

Subgraph 的延迟与一致性

Arbitrum subgraph 通常比链上最新区块落后 1–5 分钟，但在网络拥塞或 The Graph 节点同步慢时可能更长。实时监控场景下，查询前可以先检查 _meta.block.number 和当前链上高度的差，超过阈值就告警。

检查 subgraph 同步状态 
META_QUERY = """
{ _meta { block { number } } }
"""

def get_subgraph_lag(subgraph_url: str, rpc_url: str) -> int:
    """返回 subgraph 落后链上多少个区块。"""
    subgraph_block = requests.post(
        subgraph_url, json={"query": META_QUERY}
    ).json()["data"]["_meta"]["block"]["number"]

    chain_block = int(
        requests.post(rpc_url, json={
            "jsonrpc": "2.0", "method": "eth_blockNumber",
            "params": [], "id": 1,
        }).json()["result"], 16
    )

    return chain_block - subgraph_block

Polymarket 实战：~10M 条记录入库

Polymarket 在 Arbitrum 上运行，成交记录通过 The Graph 的子图暴露。批量拉取 10M 条记录时几个关键点：

  • The Graph 单次查询 first 上限是 1000，必须循环分页
  • Arbitrum 上的 block time 约 0.25 秒，所以 ID 排序大致等于时间排序，但不完全精确——同一区块里的多笔交易 ID 不连续
  • 入库前先落 S3 Parquet（按天分区），再批量写 ClickHouse；避免网络中断导致重复从头拉取
  • ClickHouse 的 ReplacingMergeTree 以 transaction hash + log index 作为唯一键，保证幂等写入

另类数据：质量参差不齐

社交和链上另类数据的工程复杂度低于 CEX 数据，但数据质量问题更突出。

LunarCrush：80M 数据点背后的问题

LunarCrush 提供 3000+ 资产的社交情绪数据（小时粒度），总量约 80M 数据点，覆盖 3 年以上历史。这个数据量本身不大，入库不是难点——难点在于数据质量：

  • 资产覆盖的幸存者偏差：当前 API 返回的是现在仍存在的资产，历史上已经下架的小币种数据不完整。做回测时要注意。
  • 社交量突变：某些时间段会有异常尖峰（可能是 bot 刷量、媒体爆炸性报道），直接用原始值做信号噪声大，通常需要做 robust normalization。
  • 历史数据重算：LunarCrush 偶尔会回填或修正历史数据，增量拉取时要设计好 upsert 逻辑。

Nansen Smart Money：成本与覆盖范围的权衡

Nansen 提供 500+ 标记钱包（机构、聪明钱、交易所等）的历史链上行为，数据深度 3 年以上。问题在于：

  • API 按查询量计费，历史数据全量拉取成本不低，评估阶段要先做样本测试
  • 钱包标签是 Nansen 自己打的，有黑盒成分；同一个地址可能同时有多个标签，使用时要明确选哪个维度
  • 覆盖的链主要是以太坊主网，Arbitrum 和 Solana 的覆盖深度较浅

跨市场数据：CME 与传统金融的接口差异

CME 的 Bitcoin/Ethereum 期货数据对量化公司有两个用途：一是研究机构资金对市场的影响（CME 的 OI 和 Coinbase 的 OI 联动），二是做 BTC CME 期货与永续合约的 basis 交易。

和 crypto 接口相比，CME 数据的获取方式更传统：通过 Databento 这类历史 tick 数据提供商按月购买，格式是 MBO（Market By Order）或 MBP（Market By Price）的压缩文件，需要用专门的 schema 解析。接口风格、数据格式、时间戳精度都和 CEX API 完全不同，不能用同一套 reader 处理。

小结：选接入方式前先问这三个问题

面对一个新数据源，先想清楚三件事：

  1. 需要多新鲜？ 分钟级用 REST 定时拉就够；秒级或更快要考虑 WebSocket 或流式接入；链上数据要接受分钟级延迟
  2. 历史数据要多深？ 交易所一般只保留最近几个月到一年的历史，更早的要找第三方归档（Databento）或接受数据不完整
  3. 数据质量如何校验？ K 线可以用成交量归零检测、相邻时间戳连续性检测；链上数据要核对 subgraph 区块高度；alt-data 要设计异常值检测
Crypto 量化的数据工程不只是「拉 K 线」——衍生品统计数据、期权波动率、链上交易记录、另类数据，每一类都有独立的接入方式和工程细节。真正的难点不是拿到数据，而是知道数据什么时候不对。
