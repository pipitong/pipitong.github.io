# 个人技术博客方案：GCP 部署 + 面向面试官的数据工作展示

## 一、GCP 部署方式（任选其一）

### 方案 A：Firebase Hosting（推荐，最简单）

- **优点**：免费额度大、HTTPS/CDN 自带、与 GCP 同生态，面试可说 "Hosted on Google Cloud (Firebase)"。
- **步骤**：
  1. 在 [Firebase Console](https://console.firebase.google.com/) 创建项目（可勾选与现有 GCP 项目关联）。
  2. 本地安装 Firebase CLI：`npm i -g firebase-tools`，登录 `firebase login`。
  3. 在博客目录执行 `firebase init hosting`，选 `public` 为当前目录或 `dist`（若用静态生成器）。
  4. 构建/复制静态文件到 `public`，执行 `firebase deploy`。
- **自定义域名**：在 Hosting 里 "添加自定义网域" 即可。

### 方案 B：Cloud Run + 静态站点

- **优点**：更偏 "基础设施"，可强调 "容器化部署在 GCP Cloud Run"。
- **步骤**：用 Nginx 或 Caddy 的 Docker 镜像托管静态文件，Dockerfile 里 COPY 站点到 `/usr/share/nginx/html`，推送镜像到 Artifact Registry，在 Cloud Run 部署该镜像。

### 方案 C：Cloud Storage + Load Balancer

- **优点**：纯静态托管、CDN，适合想讲 "对象存储 + 全球加速" 的叙事。
- **步骤**：把站点上传到 GCS bucket，开启 "网站配置"，配合 Load Balancer + HTTPS。

**建议**：先用 **Firebase Hosting** 上线，后续若想加深 "从零搭建" 的故事，可再补一篇《我把博客从 Firebase 迁到 Cloud Run 的实践》技术文。

## 二、博客内容结构（面向面试官）

### 1. 首页

- **一句话定位**：例如 "Data Engineer | 量化数据管道与基础设施"。
- **近期亮点**：半年量化数据工作 3–5 条 bullet（多交易所入湖、Prefect 编排、ClickHouse、限流与可靠性）。
- **技术文章**：列表 + 摘要，按时间或主题分类。

### 2. 技术文章（核心）

建议先写 3–5 篇，与简历/面试表述一致：

| 主题 | 面试价值 | 可写要点 |
|------|----------|----------|
| **从零为一家公司搭建线上数据管道** | 直接对应 "从零搭建" | 需求与目标 → 数据源与频率 → 架构选型（调度/存储）→ 表设计与分区 → 限流/重试/监控 → 上线与迭代。可配一张架构图。 |
| **多交易所数据采集：统一抽象与限流** | 对应 "多数据源、rate limit" | Reader/Endpoint 抽象、各交易所 API 差异、统一限流与重试策略、Prefect flow 设计。 |
| **ClickHouse 在量化数据场景的实践** | 存储与查询 | 表结构、分区与 TTL、写入批量化、简单查询示例与注意点。 |
| **用 Prefect 做数据流编排与可观测性** | 调度与运维 | DAG 设计、任务依赖、日志与状态监控、失败告警与重跑。 |
| **数据管道中的容错与一致性** | 可靠性 | 重试与 backoff、幂等写入、离线补数与在线流的一致性。 |

### 3. 技术栈 / 工具箱

- 一页列出：**调度**（Prefect）、**存储**（ClickHouse）、**对象存储**（S3/兼容）、**容器**（Docker）、**语言**（Python）等。
- 每项 1–2 句说明在你项目里的用途，方便面试官扫一眼就知道你用过什么。

### 4. 其他可浏览内容（增加停留与印象）

- **About**：简短背景（南邮 → 微软 → 量化 DE）、当前方向、联系方式（LinkedIn / 邮箱）。
- **Projects**：1–3 个核心项目卡片（如 "多交易所数据入湖管道""链上/宏观数据接入"），每项 2–3 句 + 技术标签。
- **Resume / CV**：放一页 "精简版简历" 或 PDF 下载链接，方便面试官保存。
- **分类/标签**：按 "数据工程""量化""从零搭建""可观测性" 等分类，便于按主题浏览。

## 三、写作与维护建议

- **与简历一致**：博客里的项目名、技术栈、时间线尽量和简历一致，避免矛盾。
- **突出结果**：每篇文章结尾可加 "带来的效果"（如延迟、完整性、可维护性），面试时可直接引用。
- **控制篇幅**：单篇 1500–3000 字即可，重点讲清楚 1–2 个点，配架构图或代码片段更佳。
- **持续更新**：哪怕每月 1 篇，也能体现你在持续沉淀，面试官翻到 "最近更新" 会加分。

## 四、面向非技术 / 跨行业面试官（跳槽向）

当你投的是「不那么 match」的岗位（如泛数据、业务侧、非量化公司），面试官可能是 HR 或业务负责人，没有交易所、K 线、ClickHouse 等背景。博客可以这样写，让他们也能看懂并记住你：

### 4.1 写作原则

| 原则 | 做法 |
|------|------|
| **一句话说清价值** | 每篇文章开头或结尾用 1 句「外行话」总结：例如「我们做的事，就是让公司能稳定、及时地拿到多交易所的行情数据，供策略和研究用。」 |
| **用类比代替术语** | 把「数据管道」说成「像自来水管道：数据从交易所/API 流进来，经过限流、校验，存到仓库，随时可查」；把「限流」说成「每个交易所规定每分钟只能问多少次，超了会封 IP，所以我们用统一排队」。 |
| **先结论再细节** | 第一段写「做了什么、带来什么效果」；后面再写技术选型、表设计等，并标注「对技术细节不感兴趣可跳过」。 |
| **每篇文末加「面试一句话」** | 例如：「面试时你可以说：我在量化公司从零参与搭建了多交易所数据入湖管道，保证数据及时、完整、可追溯，并用统一限流和调度避免被封禁和数据缺口。」 |

### 4.2 建议的「入门篇」文章（给毫无背景的人看）

- **《量化公司里的数据工程师在做什么？》**  
  用 2～3 个类比讲清楚：量化公司需要什么数据、数据从哪来、数据工程师负责「接进来、存好、不丢、能查」。不写代码，不堆交易所名，结尾落到「这类能力在别的行业也能复用：多数据源、实时/准实时、可靠性与可观测性」。

### 4.3 技术文章里的「友好」元素

- 在技术文章开头加一个 **「给招聘方 / 非技术面试官」** 的小框：2～3 行概括这篇文章对应简历哪一条、解决什么问题、一句话结论。
- 技术栈列表旁加一句说明：例如「Prefect：我们用它来定时、有序地拉取数据，类似一个不会乱、可监控的闹钟」。
