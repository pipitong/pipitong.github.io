## 二、从零搭建「线上数据」的流程与要点（可作文章骨架）

下面是一个可直接用于博客或面试叙述的 **从零搭建线上数据** 的流程清单；每一条都可以展开成 1～2 段。

1. **需求与目标**  
   - 业务要什么：多交易所 K 线/资金费率/持仓量/链上等；更新频率（实时/分钟级/小时级）；历史深度与一致性。  
   - 目标概括：**多源、及时、完整、可追溯**。

2. **数据源与频率**  
   - 列出数据源（Binance、Bybit、OKX、Bitget、Gate、KuCoin、Coinbase、Polymarket 等）。  
   - 每个源的 rate limit、推荐拉取间隔、数据量级 → 决定调度与存储规划。  
   - 统一抽象：**数据源 + endpoint + 更新频率**，便于编排。

3. **架构选型**  
   - **调度**：Prefect（或类似）— 每个 endpoint 一个 flow，cron/interval 调度，限流与独立重跑。  
   - **存储**：ClickHouse 作分析存储；原始数据可先落 S3 再入库，或直接写库，视延迟与成本权衡。  
   - **离线历史**：API → S3 (Parquet) → ClickHouse，与在线流 schema 一致、幂等写入。

4. **表设计与分区**  
   - 按业务域建表（kline、funding_rate、open_interest 等）。  
   - 主键与分区键：时间 + 交易所/交易对；TTL 与保留策略，控制存储膨胀。

5. **限流、重试与一致性**  
   - **限流**：各交易所有 weight/次数的限制，用统一限流（如 Redis 滑动窗口、全局 weight 池）避免 over-quota 导致封禁或数据缺口。  
   - **重试与 backoff**：拉取与写入都支持重试，保证「能拉到就不丢」。  
   - **幂等**：同一时间范围重复写入不重复计数，便于补数。

6. **交易对/符号生命周期**  
   - 交易所会上下架交易对；用 Symbol Manager 从 API 同步到 YAML+Git，PR + Slack 通知，下架需审批（如 delist 表），避免误删。

7. **上线与迭代**  
   - 先上核心数据源与核心表，再扩展；历史用离线任务补全。  
   - 迭代：更多数据源、更细监控、数据质量校验与告警。

面试时可按「需求 → 数据源 → 架构 → 表设计 → 限流与可靠性 → 上线与迭代」这条线讲，并穿插 learning_proj 里的具体数字（交易所数、数据量、延迟、代理数等）。

## 三、量化行业数据工作的「基本要素」（给外行面试官看）

用 **非术语** 的方式概括，方便你写「入门篇」或向非技术面试官解释：

- **量化公司需要什么数据？**  
  行情（价格、成交量）、衍生品数据（资金费率、持仓量、多档多空比）、期权波动率、链上/另类数据等；来自多家交易所、多种 API，格式和限制各不相同。

- **数据从哪来？**  
  交易所公开 API、第三方数据商、链上索引（如 The Graph）。每个源有「每分钟最多问几次」等限制，超了会封 IP 或限流。

- **数据工程师在干什么？**  
  把多源、多格式的数据 **接进来**（统一抽象、限流、重试）、**存好**（分区、去重、可回溯）、**不丢**（监控、告警、补数）、**能查**（表设计、查询性能）。  
  一句话：**让策略和研究团队随时能用上完整、及时、可靠的数据。**

- **和「一般互联网数据」的共通点**  
  多数据源、实时/准实时、可靠性、可观测性、从零搭管道——这些能力在电商、广告、风控等场景同样需要，面试时可强调「可迁移」。

## 四、让毫无背景的面试官理解的写作技巧

1. **每篇文章开头或结尾有一句「外行总结」**  
   例：「这篇文章讲的是：我们如何从零给公司搭了一套多交易所数据入湖管道，保证数据及时、完整，并且用统一限流避免被交易所封禁。」

2. **用类比**  
   - 数据管道 ≈ 自来水管道（数据从源头流进来，经限流、校验，存到仓库）。  
   - 限流 ≈ 排队取号（交易所规定每分钟只能问 N 次，我们统一排队，不超限）。  
   - 调度（Prefect）≈ 可监控的闹钟（到点拉数据、失败可重试、有日志可查）。

3. **技术文章加「给招聘方」小框**  
   - 2～3 行：对应简历哪一条、解决什么问题、一句话结论。  
   - 技术细节段落可加「对技术细节不感兴趣可跳过」。

4. **首页 About / 技术栈**  
   - 写一句「本博客也面向非技术面试官：部分文章用通俗语言解释数据工程与量化数据工作。」  
   - 技术栈每项 1 句用途（如 Prefect：定时、有序拉取数据，带监控与重试）。

5. **留联系方式与简历**  
   - 首页或 About：邮箱、LinkedIn、Resume/CV 下载链接，方便面试官保存。

1. **《量化公司里的数据工程师在做什么？》**（通俗入门，无背景也能懂）

4. **《ClickHouse 在量化数据场景的实践》**（存储与查询）
